# Thai Law RAG — ประมวลกฎหมายที่ดิน พ.ศ. ๒๔๙๗

ระบบถาม-ตอบกฎหมายไทยแบบ Hybrid RAG บนเอกสารจากสำนักงานคณะกรรมการกฤษฎีกา
ตอบพร้อมอ้างอิงเลขมาตราและหน้าเสมอ — ไม่ตอบจากความรู้นอกเอกสาร

> **วัดผลได้ 29/30 (97%)** จากชุดคำถามมาตรฐาน 30 ข้อ (ง่าย/ปานกลาง/ยาก อย่างละ 10) นิ่งทุกรอบ

## โจทย์ที่ยากกว่าที่คิด

ประมวลกฎหมายที่ดินไม่ใช่เอกสารเดียว แต่คือกฎหมายเล่มเดียวที่มี **53 สถานะเวลา**
(ฉบับดั้งเดิม พ.ศ. ๒๔๙๗ + ฉบับแก้ไข ๑๕ ฉบับ + ฉบับรวมสะสม ๒๐ ช่วงเวลา)

ความยากไม่ใช่ "หาไม่เจอ" แต่คือ **"เจอหลายฉบับแล้วต้องเลือกฉบับที่ใช้บังคับอยู่จริง"** —
ถามว่า _"ที่ดินใบจองห้ามโอนกี่ปี"_ ระบบเจอคำตอบใน ๕ ฉบับ ฉบับเก่าบอก ๕ ปี ฉบับปัจจุบันบอก ๑๐ ปี
เลือกผิด = ตอบด้วยกฎหมายที่ยกเลิกไปแล้ว

**แนวคิดหลัก — ข้อเท็จจริงให้โค้ดคำนวณ การเรียบเรียงให้ LLM**
งานกฎหมายรับ "ส่วนใหญ่ถูก" ไม่ได้ สายการแก้ไขกฎหมายจึงอ่านจากตัวบทด้วยโค้ด (ถูก ๑๐๐% ตรวจสอบได้
ชี้ได้ว่ามาจากไฟล์ไหน) ไม่ให้ LLM เดา ส่วน LLM มีหน้าที่เรียบเรียงคำตอบจากข้อเท็จจริงที่คำนวณมาแล้วเท่านั้น

## สถาปัตยกรรม

```
PDF ──PyMuPDF──> chunk ตามขอบ "มาตรา/ข้อ" ──> Chroma (semantic)
  └─ หน้าสแกน ──EasyOCR──┘                 └─> BM25 (ตัดคำไทย newmm)

ถาม ──> rewrite คำถามต่อเนื่อง ──> เลือกกลุ่ม (LLM router) ──> ขยายคำค้น 3 มุม
     ──> hybrid retrieve (RRF) ──> ยุบเวอร์ชันซ้ำ ──> rerank ──> LLM ตอบ
```

| ไฟล์ | หน้าที่ |
|---|---|
| `rag.py` | เครื่องยนต์: โหลด/chunk/embed/retrieve/rerank + prompt |
| `service.py` | pipeline ระดับแอป (routing, multi-turn) — ไม่ผูก UI |
| `app.py` | Streamlit UI |
| `main.py` | FastAPI |
| `ocr.py` | OCR ไทยแบบ local (EasyOCR + PyMuPDF) |

### จุดที่ต้องระวัง

- **ห้ามใช้ pdfplumber คัดข้อความไทย** — มันเรียงตาม x-coordinate ทำให้สระ/วรรณยุกต์
  หลุดตำแหน่ง (`"ทรพั ยส์ นิ"` แทน `"ทรัพย์สิน"`) พังทั้ง embedding, BM25 และตัวบทที่ LLM คัดไปตอบ
- **เลขไทย ๐-๙** ถูก normalize เป็นอารบิกตอน tokenize → ถาม "มาตรา 9" เจอ "มาตรา ๙"
- เปลี่ยน `CHUNK_SIZE` / `EMBED_MODEL` ต้อง rebuild: `python -c "import rag; rag.build_vectorstore(force=True)"`
- reranker ใช้ GPU ตัวเดียว → FastAPI ต้องรัน **worker เดียว** (อย่าใช้ `--workers >1`)

## metadata ที่เก็บต่อ chunk

ทุก chunk ถูกประทับข้อมูลประจำตัว **ตั้งแต่ตอนสร้าง** — ระบบอ่านทีละไฟล์ ก้อนที่เกิดจากไฟล์ไหน
ได้ข้อมูลของไฟล์นั้นทันทีในลูปเดียวกัน (ไม่ใช่เดาย้อนหลัง) แล้วเก็บลง Chroma คู่กับ vector
ป้ายพวกนี้คือฟิลเตอร์ที่ทำให้ตอบถูกฉบับ: **สิ่งที่ติดตอนเข้า = สิ่งที่ใช้กรองตอนออก**

รหัสประจำก้อน: `id = ชื่อไฟล์::ลำดับ` (เช่น `LandCode2497_Update-vlast_TruePDF.pdf::0003`)
— เห็น id เดียวก็ย้อนถึงต้นทางได้ + metadata อีก 18 ฟิลด์ (`rag._chroma_metadata`):

| กลุ่ม | ฟิลด์ | เก็บอะไร |
|---|---|---|
| **ระบุที่มา** | `source` · `doc_label` | ชื่อไฟล์ · ชื่อเอกสารอ่านง่าย (ใช้แสดงอ้างอิงท้ายคำตอบ) |
| | `page_start` · `page_end` | เนื้อหานี้อยู่หน้าไหนถึงหน้าไหนของไฟล์ |
| | `kind` · `version` · `year` | main/amend/update · เลขฉบับ (vlast→999) · ปีเอกสาร |
| **กรอง/จัดอันดับ** | `in_force` | ⭐ ตัวบทนี้ใช้บังคับอยู่จริงไหม — คำถามปกติกรองด้วยฟิลด์นี้ กันกฎหมายที่ยกเลิกแล้วหลุดมาตอบ |
| | `as_of_year` | ตัวบทสะท้อนกฎหมาย ณ ปีไหน (ตอบคำถามย้อนเวลา) |
| | `group` · `amend_no` | กลุ่มเอกสาร 4 กลุ่ม · รวมการแก้ถึงฉบับที่เท่าไร |
| | `article` | **มาตราหลัก**ของก้อน — ค่าเดียว ใช้จัดอันดับ (ถามเจาะมาตราไหน ก้อนที่ว่าด้วยมาตรานั้นขึ้นก่อน) |
| | `article_nums` | **ทุกมาตราที่โผล่**ในก้อน เช่น `\|8\|9\|10\|11\|` — ใช้ค้นหา (ถามมาตรา ๘ ก้อนนี้ต้องถูกเจอแม้เรื่องหลักคือ ๑๑) |
| **เสริม** | `articles` · `refs` · `section` | ป้ายมาตราแบบเต็ม · มาตราที่ถูกอ้างถึง · หมวด/ภาค |
| | `n_articles` · `is_scan` | จำนวนมาตราในก้อน · มาจาก OCR ไหม (ถูกดันท้ายแถวเพราะเชื่อถือได้น้อยกว่า) |

### ก้อน "กลาง ๆ" ที่ไม่มีหัวมาตรา รู้ได้อย่างไรว่าเป็นมาตราอะไร

98 จาก 106 ก้อนของไฟล์หลักขึ้นต้นกลางความ ไม่มีคำว่า "มาตรา" นำหน้า — ระบบจึงแท็กเป็น**รายบรรทัด**
ตั้งแต่ตอนอ่าน: มีตัวแปรจำว่าบรรทัดนี้อยู่ใต้หัวมาตราไหน ทุกบรรทัดได้ป้ายของตัวเอง พอตัดก้อน:

- `article` (ป้ายหลัก) = มาตราที่**กินเนื้อที่มากสุด**ในก้อน (`dominant_article`)
  ⚠️ ห้ามใช้ "มาตราล่าสุดที่เจอ" — เคยใช้แล้ววัดได้ป้ายผิดถึง 81% เพราะก้อนเดียวคาบหลายมาตรา
- `article_nums` = ทุกมาตราที่โผล่ในก้อน
- ตอน overlap ท้ายก้อนเก่าไปหัวก้อนใหม่ ป้ายบรรทัดถูกยกตามไปด้วย เนื้อความที่ติดมาจึงไม่ลืมสังกัด
- ป้ายรายบรรทัดเป็นนั่งร้านชั่วคราว ใช้สรุปแล้วทิ้ง — **1 chunk มี metadata ชุดเดียวเสมอ**

## เอกสารใน `data/`

ชื่อไฟล์เป็นตัวกำหนด group/metadata เอง (ไม่ต้องแยกโฟลเดอร์) — ดู `rag.parse_doc_name()`

```
LandCode2497_<Kind>-v<Ver>_<True|Img>PDF.pdf
```

| Kind | group ที่ได้ | ค้นเมื่อไร |
|---|---|---|
| `Update-vlast` | ฉบับใช้บังคับปัจจุบัน | 🟢 **ค่าเริ่มต้น** — ทุกคำถามที่ไม่ระบุปี |
| `Update-v1..v20` | ฉบับย้อนหลังตามช่วงเวลา | 🔒 เมื่อคำถามระบุ พ.ศ. |
| `Amend-v1..v15` | ประวัติการแก้ไข | 🔒 เมื่อถามว่าแก้อะไร/เมื่อไร |
| `Main-v0` | ฉบับดั้งเดิม พ.ศ. ๒๔๙๗ | 🔒 เมื่อถามตัวบทดั้งเดิม |

`TruePDF` = คัดข้อความได้ · `ImgPDF` = สแกน → เข้า OCR ภาษาไทยอัตโนมัติ (local, ดู `ocr.py`)

### ⚖️ ทำไม default ต้องเป็น `vlast` ไฟล์เดียว

กฤษฎีกาไล่ปรับตัวบทให้ทีละมาตราแล้วในฉบับรวมสะสม เทียบทั้งฉบับกับ `Main-v0` ได้ว่า:

```
เหมือนเป๊ะกับฉบับหลัก (ไม่เคยถูกแก้)   44 มาตรา
เนื้อหาต่าง (ถูกแก้ไปแล้ว)             57 มาตรา
มาตราที่เพิ่มใหม่ (๘ ทวิ, ๙/๑ ฯลฯ)     33 มาตรา
ตกหล่น                                  0 มาตรา
```

`vlast` จึงครอบคลุมทั้ง *"มาตราที่ไม่เคยแก้ → ใช้ของหลัก"* และ *"มาตราที่แก้แล้ว → ใช้ของใหม่"* อยู่ในไฟล์เดียว

ไฟล์ที่เหลือมี **ตัวบทที่ถูกยกเลิกไปแล้วปนอยู่** (เช่น `มาตรา ๙๗` ใน `Main-v0` เป็นเวอร์ชันก่อนแก้)
ถ้าค้นปนกันเมื่อไร = มีโอกาสตอบด้วยกฎหมายที่เลิกใช้แล้ว → `retrieve()` จึงล็อกไว้ที่กลุ่มที่ใช้บังคับเสมอ
การปลดล็อกตัดสิน**ด้วยกฎในโค้ด** (`service.pick_groups`) ไม่ใช่ให้ LLM เดา

นอกจากนี้ยังยุบสำเนาซ้ำ (`_dedupe_versions` — 29% ของดัชนี) และดัน chunk จาก OCR ท้ายแถว (`_demote_scans`)

## การวัดผลและการทดลอง

ทุกการตัดสินใจมีตัวเลขรองรับ ไม่ใช่เดา:

- **ชุดวัดผล 30 ข้อ** (`eval/run_eval.py`) — ให้คะแนนอัตโนมัติแบบ key-fact matching, รัน 3 รอบทุกครั้ง
  เพราะ LLM ให้ผลไม่คงที่ 100% แล้วดูค่าเฉลี่ย ปัจจุบัน **29/30 นิ่งทุกรอบ** (ตกข้อเดียวเพราะเฉลยกว้างกว่าคำถาม)
- **smoke test 32 ข้อ** (`eval/smoke_test.py`) — ตรวจกติกาที่ห้ามพังโดยไม่เรียก LLM (เร็ว/ฟรี รันก่อน commit ทุกครั้ง)
- **ทดลองแล้วเลือกไม่เปลี่ยน** (`eval/embed_compare.py`) — ลองสลับ embedding เป็นรุ่นใหม่ 2 ตัว
  วัดจริงแล้วของเดิมยังดีกว่าและนิ่งกว่า จึงไม่เปลี่ยน — negative result ที่มีข้อมูลรองรับ
- ทุก run ส่งขึ้น **MLflow** เพื่อเทียบข้ามรอบได้

## ติดตั้ง

```powershell
python -m venv .venv
.venv\Scripts\activate

# torch ต้องเป็น CUDA build ถ้ามี GPU (reranker + OCR เร็วขึ้นหลายเท่า)
pip install torch torchvision --index-url https://download.pytorch.org/whl/cu124
pip install -r requirements.txt pythainlp
```

คัดลอก `.env.example` เป็น `.env` แล้วเติม endpoint + API key จริง:

```
LLM_BASE_URL=http://your-llm-endpoint/v1
LLM_API_KEY=sk-xxxxxxxx
```

| คำสั่ง | ได้อะไร |
|---|---|

⚠️ รอบแรกช้า (~สิบนาที) เพราะต้องอ่าน PDF ทุกไฟล์ + OCR หน้าสแกน + สร้าง embedding
รอบถัดไปอ่านจาก `chroma_db/` กับ `ocr_cache/` ทันที
